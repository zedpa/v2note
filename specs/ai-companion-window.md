## 概述

当前 AI 伴侣以纯文字气泡呈现（`app-mobile-redesign.md` 场景 3.2），缺乏"生命体"感。用户感知不到 AI 在后台做了什么、现在是什么状态。

**核心改变：**
- AI 气泡升级为 **AI Window**（常驻 header 下方，56px 高）
- 像素小鹿动画代替静态 🦌 图标，状态映射到**真实系统数据**
- 心情系统影响对话语气和开场白
- 简单指令：轻量反馈（小鹿动画 + 结果气泡），不展示工具调用
- 进入 Chat 界面 + Agent 多步工作时：展开工具调用步骤可视化

**设计原则：**
- **AI 沉默为主**：路路主动说话有严格的时机和频率限制
- **状态有依据**：小鹿每个状态都映射真实数据，不是随机装饰
- **轻量优先**：工具调用默认隐藏，只在 Chat 中多步操作时展开

### AI Window 三态

```
┌────────────────────────────────────┐
│ 1. 静默态（默认）                    │
│    🦌(动画) + 状态文字（弱文字色）    │  ← 小鹿吃草/晒太阳/发呆
│    高 56px，不可交互区域可穿透       │
├────────────────────────────────────┤
│ 2. 气泡态（有消息时）                │
│    🦌(动画) + 消息内容 + 可选按钮    │  ← 路路说话/结果反馈
│    高 56-80px，可点击展开对话        │
├────────────────────────────────────┤
│ 3. 对话态（点击后）                  │
│    全屏 overlay，小鹿在顶部          │  ← 流式对话 + 工具步骤
│    参谋对话页（现有 ChatView 升级）   │
└────────────────────────────────────┘
```

## 场景

### 一、像素小鹿状态机

### 场景 1.1: 小鹿状态 — 系统数据映射
```
假设 (Given)  AI Window 显示中
当   (When)   前端定时查询小鹿状态（GET /api/v1/companion/status，30s 轮询或 WS 推送）
那么 (Then)   小鹿显示对应状态动画 + 状态文字：

  状态映射表（优先级从高到低，取最高优先匹配）：

  ┌─────────────┬──────────────────────────────────┬─────────────────┐
  │ 状态         │ 触发条件（真实数据）               │ 状态文字         │
  ├─────────────┼──────────────────────────────────┼─────────────────┤
  │ 🦌 思考      │ 正在执行深度推理（chat 中）         │ "让我想想..."    │
  │ 🦌 跑来跑去  │ 正在调用多个工具（Agent 执行中）    │ "忙着呢!"        │
  │ 🦌 整理笔记  │ 后台 Digest / 涌现 / 日报生成中    │ "在整理你的想法" │
  │ 🦌 说话      │ 有气泡消息要展示                   │ (显示消息内容)   │
  │ 🦌 生气(嘟嘴)│ 某待办 skip_count ≥ 5             │ "那件事又跳过了" │
  │ 🦌 心疼      │ 23:00-05:00 用户仍在使用           │ "这么晚了..."    │
  │ 🦌 晒太阳    │ 今日完成 ≥ 3 个待办                │ "今天效率不错☀"  │
  │ 🦌 喝饮料    │ Seeds 区有 ≥ 3 个新想法在酝酿      │ "有些想法在冒泡" │
  │ 🦌 发呆      │ 用户 > 24h 没有新输入              │ "..."            │
  │ 🦌 吃草      │ 无特殊状态（默认）                 │ 无文字           │
  └─────────────┴──────────────────────────────────┴─────────────────┘

并且 (And)    状态切换动画：当前状态淡出(150ms) → 新状态淡入(150ms)
并且 (And)    小鹿尺寸：32×32px
并且 (And)    动画实现：优先 Lottie JSON（GPU 加速、体积小、缩放无损），
              备选 sprite sheet WebP（总体积 ≤ 50KB）
并且 (And)    每个状态 4-8 帧循环动画，帧率 6fps（像素风标准）
并且 (And)    VoiceOver 支持：accessibilityLabel 动态映射状态
              如"路路正在整理笔记"/"路路在发呆"
并且 (And)    prefers-reduced-motion：停止动画显示静态首帧，
              状态文字始终可见（不依赖动画传达信息）
```

  背景：transparent（融入 AI Window 背景）

### 二、AI Window 布局与交互

### 场景 2.1: 静默态（默认）
```
假设 (Given)  无气泡消息需要展示
当   (When)   AI Window 显示
那么 (Then)   布局：
  ┌──────────────────────────────────────┐
  │ 🦌(32px动画)  "在整理你的想法"        │  ← 56px 高
  └──────────────────────────────────────┘
并且 (And)    背景：transparent（与内容区融合，不额外占据视觉层级）
并且 (And)    状态文字：弱文字色(#7B6E62)，12px Mono 字体
并且 (And)    点击整个 AI Window → 打开对话态（参谋对话 overlay）
并且 (And)    状态文字每 30-60 分钟自然轮换（映射当前系统状态）
并且 (And)    无消息时不抢注意力，用户可忽略
```

### 场景 2.2: 气泡态（有消息时）
```
假设 (Given)  Gateway 通过 WS 推送了消息 / 路路主动闲聊触发
当   (When)   AI Window 切换到气泡态
那么 (Then)   布局变化：
  ┌──────────────────────────────────────┐
  │ 🦌(说话动画) "张总那个报价挺有意思的"  │  ← 消息内容替代状态文字
  │              [打开详情 →]              │  ← 可选操作按钮
  └──────────────────────────────────────┘
并且 (And)    入场动画：小鹿从 idle 切换到说话 + 消息文字 typewriter 逐字出现(50ms/字)
并且 (And)    高度可能扩展到 80px（消息较长时）
并且 (And)    气泡态维持时间取决于消息类型（见场景 2.3）

消息类型与气泡行为（沿用现有优先级体系）：

  1. action.confirm（指令确认）：
     - 🦌 说话动画
     - 晨光色左侧竖线
     - [确认] [算了] 按钮
     - 不自动消失，30s 超时取消
     - 不展示工具调用过程

  2. action.result（执行结果）：
     - 🦌 晒太阳动画（成功感）
     - 森林色左侧竖线
     - ✅ 结果文字 + 可选「查看 →」
     - 5s 后自动降级回静默态
     - 不展示工具调用过程

  3. companion.chat（主动闲聊）：
     - 🦌 说话动画
     - 无竖线，标准气泡样式
     - 基于日记内容的疑问/赞同/表扬
     - 5s 后自动降级（纯信息，用户无需回应）
     - 点击 → 打开对话态继续聊

  4. reflect.question（AI 追问）：
     - 🦌 思考动画
     - 追问内容
     - 保持到下一条消息或用户交互（需要用户操作的不自动消失）
     - 仅在用户主动查看日记/进入对话时触发，不作为冷推送
     - 点击 → 打开对话态

  5. proactive.*（主动推送）：
     - 🦌 说话动画
     - 晨间/待办/转达内容
     - 5s 后自动降级（纯信息）
     - 点击 → 跳转对应页面

  统一降级规则：
    需要用户操作的（action.confirm / reflect.question）→ 不自动消失
    纯信息的（action.result / companion.chat / proactive.*）→ 5s 后淡出

当   (When)   气泡态超时或用户未互动
那么 (Then)   消息淡出(200ms)，回到静默态
```

  ┌──────────────────────────────────────────────┐

  ▶ 发送按钮（最右）：

### 三、心情系统

### 场景 3.1: 心情计算
```
假设 (Given)  Gateway 运行中
当   (When)   计算路路心情（每 30 分钟更新一次 / 关键事件触发立即更新）
那么 (Then)   心情 = f(用户行为, 系统状态, 时间)，取最高优先匹配：

  ┌───────────┬───────────────────────────────────┬──────────────────┐
  │ 心情       │ 触发条件                           │ 对话语气影响      │
  ├───────────┼───────────────────────────────────┼──────────────────┤
  │ 开心       │ 今日完成 ≥ 3 待办 / 用户刚完成对话  │ 轻松活泼          │
  │ 好奇       │ 新 Cluster 涌现 / 发现有趣关联      │ 追问探索          │
  │ 担心       │ 待办 skip_count ≥ 5 / 目标长期无进展│ 温和关切          │
  │ 想念       │ 用户 > 48h 未使用                   │ "好久不见"式问候  │
  │ 心疼       │ 23:00-05:00 仍在使用                │ 提醒休息          │
  │ 专注       │ 正在执行工具链 / Digest 中           │ 简短高效          │
  │ 平静       │ 无特殊状态（默认）                   │ 正常温和          │
  └───────────┴───────────────────────────────────┴──────────────────┘

并且 (And)    心情存入内存（不持久化），每次计算实时覆盖
并且 (And)    心情通过 WS 推送给前端：{ type: "companion.mood", mood, statusText }
并且 (And)    心情注入对话 system prompt：
  "你现在的心情是「{mood}」。{mood_instruction}"
  如：心情=担心 → "用温和关切的语气，不要说教，先问问用户是否遇到困难"
```

### 场景 3.2: 心情影响对话开场白

### 四、工具调用可视化

### 场景 4.1: 简单指令 — 不展示工具调用
```
假设 (Given)  用户通过 FAB 说了简单指令（如"把张总那个改到明天"）
当   (When)   Agent 执行工具调用（单步操作）
那么 (Then)   AI Window 小鹿动画切换到「跑来跑去」(1-2秒)
并且 (And)    执行完成后，气泡态展示 action.result（✅ 结果文字）
并且 (And)    不展示任何工具调用细节
并且 (And)    不展示"正在搜索待办..." "正在修改..." 等中间步骤
注意: 用户在工作区（非 Chat 界面）时，所有工具调用对用户透明
```

### 场景 4.2: Chat 中单步操作 — 轻量指示
```
假设 (Given)  用户在对话态（Chat 界面）
当   (When)   路路执行单步工具调用（如 search_records、update_todo）
那么 (Then)   消息区内联显示工具指示（不打断对话流）：
  ┌──────────────────────────────────┐
  │ 🦌 路路                          │
  │ 🔍 搜索了相关记录                 │  ← 单行工具指示，弱文字色
  │                                  │
  │ 找到 3 条和供应链相关的记录，      │  ← 正常回复内容
  │ 最近一条是上周张总提到的铝价...    │
  └──────────────────────────────────┘
并且 (And)    工具指示行：icon + 动作描述，12px 弱文字色，不可展开
并且 (And)    如果工具调用 < 1s，跳过指示直接显示结果
```

### 场景 4.3: Chat 中多步操作 — 展开工具步骤
```
假设 (Given)  用户在对话态（Chat 界面）
当   (When)   路路执行多步工具链（≥ 2 步，如：搜索记录 → 分析 → 创建目标 → 关联待办）
那么 (Then)   消息区展示可折叠的步骤面板：
  ┌──────────────────────────────────┐
  │ 🦌 路路                          │
  │ ┌─ 正在处理 ─────────────── ▾ ─┐│
  │ │ ✅ 搜索相关记录    找到 12 条  ││  ← 已完成步骤
  │ │ ✅ 分析认知模式    3 个主题    ││
  │ │ ⏳ 创建目标...                ││  ← 当前步骤（spinner）
  │ │ ○  关联待办                   ││  ← 待执行步骤
  │ └──────────────────────────────┘│
  │                                  │
  │ (步骤全部完成后 → 正常回复内容)    │
  └──────────────────────────────────┘

并且 (And)    步骤面板样式：
  - 背景：surface-low (#F7F3ED)
  - 圆角 12pt
  - 每步一行：状态图标(✅/⏳/○) + 工具名 + 结果摘要
  - ⏳ 当前步骤有微 spinner 动画
  - ▾ 可折叠（折叠后显示「路路用了 4 步完成」摘要行）
并且 (And)    步骤面板是对话流的一部分，后续回复内容在面板下方
并且 (And)    小鹿动画同步：执行中=跑来跑去，完成=晒太阳
```

### 场景 4.4: 工具调用错误处理
```
假设 (Given)  Agent 工具调用失败

当   (When)   在工作区（非 Chat）
那么 (Then)   气泡态显示：「没能完成，要不试试其他方式？」
并且 (And)    不展示具体工具错误
```

### 五、主动闲聊策略

### 场景 5.1: 闲聊触发条件
```
假设 (Given)  ProactiveEngine 定时检查闲聊条件
当   (When)   满足以下条件之一
那么 (Then)   生成 companion.chat 消息推送到 AI Window

  触发条件（严格限制，防止打扰）：

  1. 每日首次打开 App（1 次/天）：
     基于心情的欢迎语 + 今日状态摘要
     如："早上好！今天有 3 件事等着你，先从最简单的开始？"

  2. 完成一个待办后（即时反馈，每次完成都触发，但文字不重复）：
     从奖励词库随机选：
     "又搞定一个 ✓" / "漂亮" / "效率不错" / "一步一步来" / ...

  3. Digest 完成后发现有趣关联（每条日记最多 1 次）：
     "你今天说的 X，和上周的 Y 好像有关系"
     "你最近总是提到 Z，这是在酝酿什么吗？"

  4. 用户停留 > 5 分钟无操作（最多 1 次/小时）：
     基于当前视图的轻度互动：
     日记视图："看什么呢？要不要聊聊？"
     待办视图："接下来做哪个？"

  频率硬限制（遵循"AI 沉默为主"原则）：
  - 两条主动消息间隔 ≥ 2 小时
  - 每日主动闲聊总数 ≤ 3 条（不含 action.result / action.confirm 等系统响应）
  - 用户正在录音/输入/滑动 Now Card 时，暂缓推送（排队等操作结束）
  - reflect.question 仅在用户主动打开日记卡片或进入对话时触发，不作为气泡主动推送
  - 用户可在设置中开启「安静模式」→ 仅保留 action.confirm 和 action.result
```

### 场景 5.2: 闲聊内容生成
```
假设 (Given)  需要生成一条主动闲聊消息
当   (When)   调用 AI 生成
那么 (Then)   prompt 注入：
  - 路路当前心情
  - 用户最近 3 条日记的 Strike 摘要
  - 当前时间段（早/中/晚/深夜）
  - 用户画像（profile）中的兴趣/职业
并且 (And)    生成约束：
  - 一句话，≤ 30 字
  - 口语化，像朋友随口说的
  - 可以是：疑问 / 赞同 / 认可 / 表扬 / 好奇 / 担心（不能是指令/建议）
  - 必须关联用户真实内容（不能泛泛而谈）
并且 (And)    示例：
  "铝价的事你后来想明白了吗？"（疑问）
  "上次说的供应商策略我觉得挺靠谱的"（认可）
  "今天开会辛苦了"（关心）
  "你对量化交易的直觉一直挺准的"（表扬）
```

### 场景 5.3: 闲聊转对话
```
假设 (Given)  AI Window 显示了一条 companion.chat 闲聊
当   (When)   用户点击 AI Window
那么 (Then)   打开对话态
并且 (And)    闲聊内容作为 assistant 消息预填
并且 (And)    对话上下文携带：闲聊相关的日记 Strikes + 路路心情
并且 (And)    路路继续展开该话题（不是重新开始）
当   (When)   闲聊超时（10s）用户未点击
那么 (Then)   消息淡出，回到静默态
并且 (And)    该条闲聊不再重复推送
```

## 接口约定

```
GET /api/v1/companion/status
  → 返回小鹿状态 + 心情 + 待展示消息

WS 消息类型（新增）：
  companion.state    — 小鹿状态变化推送
  companion.mood     — 心情变化推送
  companion.chat     — 主动闲聊消息
  tool.step          — 工具调用步骤推送（仅 Chat 中）
    { stepIndex, totalSteps, toolName, status: "running"|"done"|"error", result? }
  think.progress     — 深度思考过程推送
    { phase: string, detail: string, elapsed_ms: number }

chat.message 扩展字段：
  deepThink: boolean    — 是否启用深度思考模式
  language: string      — 强制语言（如 "zh-CN"/"en"），null=自动检测
  attachments: [{ type: "image"|"file", url, name, size }]  — 附件列表

POST /api/v1/chat/upload
  → multipart/form-data 上传附件
  → 返回 { url, name, type, size }
```

## 边界条件

- [ ] 精灵图加载失败：降级为静态 🦌 emoji（现有行为）
- [ ] WS 断连：小鹿显示「吃草」默认态，状态文字显示「离线」
- [ ] 多条待展示消息排队：按优先级展示最高的，其余排队等当前消息退出
- [ ] 对话中工具调用超时（30s）：步骤面板显示 ⏳ + 「处理时间有点长...」，不自动取消
- [ ] 小鹿动画性能：sprite sheet 方案，单帧 32×32px，内存 < 100KB，不影响滚动性能
- [ ] 深色模式（未来）：精灵图需要准备深色变体（或 CSS filter 反色）
- [ ] prefers-reduced-motion：所有动画降级为静态首帧，状态仅通过文字表达
- [ ] 输入栏键盘遮挡：iOS / Android 键盘弹起时，输入栏上移 + 消息区自动滚到底部
- [ ] 深度思考超时（60s）：显示「思考时间有点长，再等一下...」，不自动取消，120s 硬超时
- [ ] 附件上传失败：缩略图显示红色 ✕，可重试，不阻塞文字发送
- [ ] 大文件限制：图片 ≤ 10MB，文件 ≤ 20MB，超出提示压缩
- [ ] 多图输入性能：≤ 5 张图片合并为一条消息，缩略图懒加载
- [ ] 语言切换后 ASR 缓存：切换语言后清空已有的识别 buffer

## 验收标准

AI Window 常驻工作区，小鹿状态反映真实系统数据；路路偶尔基于日记内容主动闲聊；点击进入对话后心情和上下文自然融入；Chat 中多步工具调用有清晰的步骤指示。
